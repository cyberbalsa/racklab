# M2.5 — Baseline Ops Smoke

**Status:** Not started.
**Estimated effort:** 1–2 weeks.
**Depends on:** M0.5, M2.
**Unblocks:** M11a (and by transitive dependency, M12 and the TLS-dependent flows).

## Goal

Prove that the Baseline profile actually works end-to-end as a deployable system before TLS, Scale, and operational milestones depend on it. M2.5 is short, deliberately scoped to "operate the Baseline you built," and it surfaces the integration bugs that only show up when web + workers + Postgres + Redis + Reverb + the bootstrap TLS cert are running together on a real host. Without M2.5, the next time anyone tries to install RackLab is when M11a needs to verify the TLS admin GUI works — too late.

## In scope

- PRD §16 Container Operations (the operational requirements: health checks, graceful shutdown, queue drain, explicit migration command, backup/restore for Baseline).
- PRD §05 Architecture (the deployable system end-to-end).
- The Baseline install path proven by M0.5.
- The deployment lifecycle proven by M2 on the fake provider.

## Dependencies

- M0.5 — `scripts/baseline-install.sh` works.
- M2 — fake provider + deployment lifecycle works in development.

## Deliverables

- **Operator runbook** (`docs/runbooks/baseline-operations.md`): step-by-step procedures for the operations an admin needs every week — start, stop, restart, drain a worker pool, check health, rotate the bootstrap TLS cert, back up Postgres (including the `broadcast_event_log` and outbox tables) + Redis snapshot + the Composer vendor dir + plugin-config, restore from backup, upgrade in place.
- **Health-check endpoints** on every container: `GET /healthz` returns 200 when ready, 503 when not. systemd `Restart=on-failure` + `ExecStartPre` checks integrate with this.
- **Graceful shutdown** on every worker: SIGTERM handler stops pulling new messages (Horizon respects `--timeout` and stops cleanly), finishes the current job up to the per-pool drain deadline, exits 0. `systemctl stop` works without losing in-flight jobs.
- **`php artisan racklab:migrate` command**: explicit, idempotent, runs core + plugin migrations in dependency order per PRD §13.
- **`php artisan racklab:backup` and `php artisan racklab:restore` commands**:
  - Backup: dumps Postgres (including the `broadcast_event_log` and outbox tables), snapshots Redis (`BGSAVE`), tars the Composer vendor dir + plugin-config + secret-backend dev backend + bootstrap TLS files; outputs to a configurable destination.
  - Restore: takes a backup archive + verifies integrity (sha256, schema-version match) + restores to a fresh install.
- **Soak test** (4-hour, not the M13d 24-hour version): continuous deployment requests against the fake provider while running periodic backup, restart, and worker-pool drain operations. No data loss, no orphaned `Job` rows, alerts (logged, no notification plugin yet) for anything unexpected.

## Acceptance criteria

- [ ] Operator runbook is the only documentation an admin needs for the week-1 operations listed; verified by a colleague-rehearsal where someone unfamiliar with RackLab installs + operates it using only the runbook.
- [ ] `systemctl restart racklab-provider-worker@1` during an active fake-provider deployment: the deployment completes successfully (graceful drain works); no `Job` row stuck in `pending` past the reconciler interval.
- [ ] `php artisan racklab:backup --to=/tmp/racklab-backup.tar.gz` produces a backup; nuking Postgres + Redis state; `php artisan racklab:restore --from=/tmp/racklab-backup.tar.gz` restores fully — every committed deployment from before the backup is queryable + the deployment state matches.
- [ ] 4-hour soak test against fake provider: zero deadlocks, zero stuck Jobs by reconciler standards, zero data loss across multiple restart + backup + drain cycles.
- [ ] Health checks correctly report not-ready during startup (database not migrated yet, Redis not connected yet, Reverb daemon not running yet) and ready once everything is up.
- [ ] Idempotent install: re-running `scripts/baseline-install.sh` against an existing install upgrades in place without data loss.

## Test layers

- **Tiny / unit** (Pest 4): `racklab:migrate` dependency-ordering logic; backup archive integrity checks (sha256 + schema-version validators); graceful-shutdown signal handler logic.
- **Contract** (Pest 4): health-check Protocol against a fake-state web tier; backup/restore against fake Postgres + Redis sources.
- **Integration** (Pest 4): drain semantics under load with testcontainers; backup → corrupt state → restore round trip.
- **Browser E2E** (Dusk, nightly): the 4-hour soak runs on real Ubuntu 24.04 in CI on a self-hosted runner.

## Risks / open questions

- **Backup encryption**: does the backup archive need to be encrypted at rest? Operators in regulated environments will want it; default is unencrypted with a documented encryption recipe; the encrypted-by-default variant lands in M13c.
- **Redis snapshot fidelity**: Redis `BGSAVE` captures in-flight queue state at a point in time. Verify that restore from a Redis snapshot leaves the Horizon queue in a consistent state (jobs not in the snapshot are recoverable via the Postgres `jobs` table and the reconciler). Test the failure modes for partial snapshots.
- **Composer vendor dir + plugin-config in the backup**: by including them, restore guarantees the same plugin set is enabled post-restore. But if a plugin's package is missing from Packagist (yanked) at restore time, what happens? Document: restore fails fast with a clear error pointing at the missing package.
- **Self-hosted CI runner for E2E**: codex called this out earlier — Quadlet + systemd E2E in GitHub Actions's hosted runners isn't a normal path. M2.5's nightly E2E requires either a self-hosted runner with systemd, a nested-VM runner action, or a direct-process E2E profile that skips systemd. Decide before M2.5 implementation.
- **The runbook is documentation**: it rots faster than code. Add a CI gate that lints the runbook for stale command references against the actual CLI surface.

## Out of scope (deferred)

- HA Postgres / HA Redis — M13a.
- Continuous backup (WAL streaming) — M13c. M2.5 is point-in-time backup only.
- Backup encryption by default — M13c.
- 24-hour soak — M13d.
- Scale-profile install runbook — M12; M2.5 is Baseline only.
- A backup-and-restore admin GUI — eventually, but M2.5 is CLI-only.
