# M0.5 — Packaging + Runtime Install

**Status:** Not started.
**Estimated effort:** 2–3 weeks.
**Depends on:** M0.
**Unblocks:** M2, M2.5, M11a/M11b, M12 (anything that assumes RackLab is actually installable and that plugins persist across container restarts).

## Goal

RackLab is installable and upgradeable in both deployment profiles, and plugins survive container restarts and image refreshes. PRD §16 ships versioned container images, so runtime plugin installation needs explicit package-store persistence; otherwise plugins disappear on the next image refresh. M0.5 closes that gap.

## In scope

- PRD §16 Container Operations (the install + upgrade story).
- PRD §13 Plugin System (the install lifecycle's persistence requirements).
- Container image build pipeline (multi-stage `Containerfile`, base-image policy, SBOM generation, license scan).
- Plugin Composer package store persistence design for Baseline (Quadlet) and Scale (Nomad).
- Dependency-audit + license-policy + plugin-package-provenance gates in CI.

## Dependencies

- M0 — the project skeleton, plugin lifecycle CLI, `composer.json` + `package.json` lockfiles.

## Deliverables

- **Container image build pipeline** (`.github/workflows/build-images.yml`):
  - Multi-stage `Containerfile` builds `racklab-web`, `racklab-provider-worker`, `racklab-script-worker`, `racklab-console-worker`, `racklab-scheduler-reconciler`, `racklab-notification-worker` from one repo.
  - Base image policy: `php:8.3-cli-alpine` or `ubuntu:24.04` (decide in M0.5 implementation; cite reasons).
  - Images pushed to `ghcr.io/cyberbalsa/racklab/*` on tag and on `main`. Pinned by git SHA + semver.
  - SBOM (CycloneDX) generated per image; attached as an artifact in CI.
  - License-policy scan rejects `GPL-3.0`/`AGPL-3.0` (and `BSL` outside the documented Nomad acceptance) in the runtime image.
- **Plugin package persistence**:
  - The plugin lifecycle persists installed Composer packages to a **named volume** mounted into every RackLab container (`/var/lib/racklab/plugins/packages/`). This is the persistent location for installed plugin packages; the running PHP environment loads from here at container start.
  - A `racklab-plugin-bootstrap` init container (Baseline as a Quadlet `Type=oneshot`; Scale as a Nomad `prestart` task) reads the persisted plugin-config + package store and runs `composer install` from the plugin manifest before the main container process starts.
  - The plugin lifecycle's `racklab plugin install` command writes to the package store + updates the persisted plugin-config file; `racklab plugin uninstall` removes both. Survives image refresh.
  - SBOM regeneration on plugin install/uninstall.
- **Baseline install script** (`scripts/baseline-install.sh`):
  - Detects systemd availability, Podman version, Postgres + Redis presence.
  - Copies Quadlets to `/etc/containers/systemd/`, runs `systemctl daemon-reload`, enables and starts the `racklab-runtime.target`.
  - Generates the self-signed bootstrap TLS cert (per M11a).
  - Creates a minimal `racklab.toml` + the persistent `racklab` Unix user.
  - Idempotent: re-running upgrades in place without data loss.

- **Installer flag surface** for `scripts/baseline-install.sh` — every interactive prompt has a matching flag so the script runs unattended in CI, cloud-image bakes, MaaS/kickstart deployments, and Ansible playbooks. The script is "interactive by default; non-interactive when flagged." Flag set:

  | Flag | Purpose |
  |---|---|
  | `--non-interactive` / `-y` | Skip every prompt; use defaults or fail fast if a required flag is missing. |
  | `--config-file=PATH` | Read flag values from a TOML / YAML file. CLI flags override file values. |
  | `--domain=FQDN` | Server hostname for the TLS cert SAN + `APP_URL` / trusted hosts config. |
  | `--listen-addr=ADDR` | Bind address (default `0.0.0.0`). |
  | `--listen-port=PORT` | Bind port (default `443`). |
  | `--data-dir=PATH` | RackLab state root (default `/var/lib/racklab`). |
  | `--admin-email=EMAIL` | First admin user; created automatically if absent. |
  | `--admin-password-stdin` | Read the initial admin password from stdin (for secret-manager piping). |
  | `--admin-token-file=PATH` | Write the initial admin token to this file after install (cloud-image baking). |
  | `--tenant-slug=SLUG` | Override the default `rit` tenant slug (the migration's default-tenant is renamed to match). |
  | `--tenant-name=NAME` | Override the default tenant display name. |
  | `--db-mode=internal\|external` | Run Postgres as a Quadlet (internal) or expect an external one. Default `internal`. |
  | `--db-url=URL` | External-mode connection string. |
  | `--redis-mode=internal\|external` | Same for Redis. Default `internal`. |
  | `--redis-url=URL` | External-mode Redis URL. |
  | `--tls-mode=self-signed\|provided\|skip` | TLS bootstrap mode. `skip` is dev-only and emits a warning. |
  | `--tls-cert=PATH` / `--tls-key=PATH` | Required when `--tls-mode=provided`. |
  | `--profile=baseline` | Future-proof — only `baseline` accepted in M0.5 (Scale uses the runbook, not this script). |
  | `--upgrade` | Upgrade-in-place mode. Refuses if RackLab is already at a newer version. |
  | `--uninstall` | Clean removal (data optionally preserved with `--uninstall --keep-data`). |
  | `--dry-run` | Print every action the script would take; perform none. |
  | `--log-format=text\|json` | Output shape. |
  | `--log-file=PATH` | Tee log to a file. Stdout still gets a summary. |
  | `--quiet` / `--verbose` | Verbosity level. |
  | `--skip-systemd-enable` | Skip `systemctl enable` (useful when the script runs inside an image build that activates units at first boot). |
  | `--accept-license` | Required in `--non-interactive` mode; acknowledges the Apache-2.0 + plugin-license-policy implications. |
  | `--version` | Print the script + RackLab version and exit. |
  | `--help` | Usage. |

  **Behaviour:** if `--non-interactive` is set and any *required-for-this-path* flag is missing (e.g., `--admin-email` without an existing admin), the script exits non-zero with a clear message naming which flag is missing. Without `--non-interactive` the script prompts interactively for missing values, with sensible defaults shown for each prompt.

- **Cloud-image bake target** (`scripts/build-cloud-image.sh` — thin wrapper around the installer + `cloud-init` user-data injection): drives `baseline-install.sh --non-interactive --config-file=cloud-image.toml --admin-token-file=/var/lib/racklab/first-login.token` and produces a qcow2 / VMDK ready for cloud providers. The token in the file is the one-time first-login credential; logging in invalidates it.

- **Ansible role wrapper** (`contrib/ansible/racklab-baseline/`): the same flags exposed as Ansible role variables. Demonstrates that the installer is automatable by configuration-management tools without the script needing to know about Ansible.
- **Scale install runbook** (`docs/runbooks/scale-install.md`):
  - Step-by-step Nomad cluster bootstrap + RackLab service registration.
  - Caddy TLS state provisioning (M11a handles the detail; M0.5 documents the shared-volume layout).
  - Shared volumes for Caddy TLS state, plugin package store, secret backend, artifact storage.
  - Upgrade-in-place: Nomad `update` stanza usage; `auto_revert` semantics.
- **CI gates added**:
  - `composer audit` against the lockfile (already named in M0 — M0.5 verifies it runs against the image's resolved environment, not just `composer.json`).
  - SBOM diff test: any new image build that changes the SBOM materially requires a SBOM diff comment on the PR.
  - License-policy regression test (snapshot of dependencies' licenses; PR fails on new GPL/AGPL/BSL outside the allowlist).
  - Plugin-package-provenance check: every package in the plugin store must have a verifiable origin (Packagist metadata for first-party; signed for institutional plugins).

## Acceptance criteria

- [ ] `scripts/baseline-install.sh` on a fresh Ubuntu 24.04 box brings RackLab up over HTTPS using the bootstrap cert in under 10 minutes from clone-to-login.
- [ ] `scripts/baseline-install.sh --non-interactive --domain=lab.example.edu --admin-email=ops@example.edu --admin-password-stdin --tls-mode=self-signed --accept-license < password.txt` on a fresh Ubuntu 24.04 box succeeds with exit code 0, creates the admin user, writes the admin token to the configured path, and the resulting RackLab instance accepts the admin login over HTTPS.
- [ ] `scripts/baseline-install.sh --non-interactive` without all the required-for-this-path flags exits with a non-zero code and an error message naming the missing flags. Re-running with the missing flags succeeds.
- [ ] `scripts/baseline-install.sh --dry-run --non-interactive --domain=lab.example.edu --admin-email=ops@example.edu --tls-mode=self-signed --accept-license` prints every action that would be taken and changes nothing on disk.
- [ ] `scripts/baseline-install.sh --upgrade` on an already-installed system pulls newer image tags, runs migrations, and keeps the existing `racklab.toml` + admin user untouched; on a system at the same version, exits with "nothing to do".
- [ ] `scripts/baseline-install.sh --uninstall --keep-data` removes Quadlets + systemd target but leaves `/var/lib/racklab` intact for a future re-install.
- [ ] After `scripts/baseline-install.sh`, installing `racklab/plugin-hello` via `racklab plugin install racklab/plugin-hello`, migrating (`racklab plugin migrate racklab/plugin-hello`), enabling, then **stopping all RackLab containers via `systemctl stop racklab-runtime.target`, refreshing the images via `podman auto-update`, and restarting** leaves the plugin installed, migrated, and enabled. The persistence story works.
- [ ] An attempt to install a plugin whose license matches a forbidden category (GPL-3.0 / AGPL-3.0 outside the allowlist) is refused at install time with a clear error pointing at the license-policy file.
- [ ] The SBOM (CycloneDX JSON) for `racklab-web` exists as a build artifact and is committed under `dist/sboms/` on tagged releases.
- [ ] `composer audit` finds zero known-vulnerable dependencies in the image's resolved environment; PR with a vulnerable dependency fails CI.
- [ ] The Scale runbook successfully provisions a 3-node Nomad cluster + a worker host + Caddy/FrankenPHP TLS (or upstream load-balancer TLS) in under 90 minutes from a documented starting state.

## Test layers

- **Tiny / unit**: `racklab.toml` config parser; license-policy matcher; SBOM diff comparator.
- **Contract**: `racklab-plugin-bootstrap` init-container behavior against a fake plugin package store; the plugin-install CLI against the package store + plugin-config write path.
- **Integration**: full install → install plugin → restart cycle on a testcontainers Ubuntu 24.04 + Podman environment.
- **E2E** (nightly): the actual `scripts/baseline-install.sh` flow on a real VM (CI provisions a fresh VM, runs the script, verifies login + plugin install + restart persistence).

## Risks / open questions

- **Plugin package store + air-gapped installs**: institutions without internet access during install need to ship the plugin package store pre-populated. Document a "bundled offline installer" build target.
- **Plugin version pinning vs floating**: when an admin runs `racklab plugin install racklab-foo`, does it pull the latest, or a pinned version per the plugin-config? Recommend: per-config-pin with `racklab plugin upgrade` as a separate command. Defines the upgrade audit trail.
- **License policy**: RackLab is Apache-2.0, so GPL-3.0 / AGPL-3.0 strictness stays high by default. GPL/AGPL/BSL dependencies need an explicit allowlist entry and documented operational/legal acceptance before they can enter runtime images or installable plugins.
- **The plugin package store on Scale-profile multi-host**: needs to be a shared volume (NFS/Ceph) so every Nomad worker sees the same packages. Document the storage requirement.
- **SBOM tool choice**: `syft` is the de facto standard for container SBOMs in 2026. Pin a version; document the upgrade policy.
- **Flag stability vs evolution**: the installer flag set is a public interface — CI / Ansible / cloud-image bakers depend on flag names not breaking between releases. Treat the flag set as a versioned contract (drop or rename a flag only on a `--version` major bump). Add new flags freely.
- **Secret materials in `--config-file`**: the config file can carry the admin password and TLS key. Document file-mode requirements (`0600`, owned by the install user) and refuse to read configs with broader permissions in `--non-interactive` mode.

## Out of scope (deferred)

- A plugin marketplace UI — operator installs by name; marketplace is post-v1.
- Cryptographic signing of first-party plugins (sigstore/cosign) — recommended for post-v1; M0.5 ships Packagist-metadata-verification only.
- Multi-architecture builds (arm64 + amd64) — amd64 only in v1; arm64 lands in M13d or later if there's demand.
- Containerless install (RackLab running directly on the host without Podman) — explicit non-goal.
